A walkthrough

cmdq

A small terminal wrapper that lets you type the next command while one is still running. This document walks through the problem, the options ruled out, the architecture, and the lessons — both Rust-specific and general — that came out of building it.

01The problem

In a normal terminal, the running command holds the TTY. So while it runs:

Invisible typing

You can't see the next command you're typing — bytes fall into the running command's stdin or smear across its output.

No edit / cancel

If you typed it wrong, you can't fix it. There's no place to fix it in.

No queueing

If you have three commands lined up, you have to babysit each finish.

before — your shell, naked
$ npm run build building... [████████░░░░░░░░] 47% npm warn deprecated foo@1.2.3 git st← goes... where? is it being typed? did 'g' just building... [██████████░░░░░░] 62% abort the build? you can't tell
The framing. The shell can't help you here — it has handed the foreground job control of the TTY. So the fix has to live above the shell, at a layer that still owns the screen and keyboard while the shell's child runs.

02Options weighed

Four serious approaches were on the table. Three are dead ends because of where they sit in the I/O stack.

ApproachWhat it triesWhy it failsVerdict
zsh / OhMyZsh plugin
shell-only		 Hook preexec/precmd; render a "type the next thing" UI from inside the shell. While a foreground command runs, the shell process is blocked. It can't read keystrokes or paint anything — the kernel has handed the TTY's foreground process group to the running command. Plugins literally cannot do this. No
readline / zle hook
line-editor		 Override the line editor to remember "second command". Same problem. zle is only alive when zsh has the prompt back. No
tmux command-prompt
multiplexer		 Use tmux's command-prompt binding to enter text. Works, but: (a) requires tmux, (b) can't inject the typed bytes back into the running pane's shell at the right moment, (c) doesn't show a queue, (d) doesn't handle conditional chaining. Partial
PTY wrapper
winning		 A separate process spawns the user's shell in a pseudo-terminal. The wrapper owns the real TTY: it draws, it reads keys. It forwards bytes to the inner shell as needed. Nothing — this is the layer that has both the screen and the keyboard. It only needs to know when the shell is at a prompt vs running a command. Yes
Reusable insight. When a layer "can't" do the thing you want, ask whether the constraint is fundamental or whether you can move up one layer. Wrapper-around-the-thing is a recurring pattern: tmux wraps shells, screen wraps shells, debuggers wrap processes, containers wrap processes, IDE terminals wrap shells. All of these exist because the wrapped layer is busy and the wrapping layer is the one that can keep responding.

03Architecture in one diagram

Every layer has one job and a clean seam to the next:

  • portable-pty spawns the user's shell and gives back a master file descriptor — bytes the shell writes, bytes you write to the shell.
  • vt100::Parser consumes those bytes and maintains a virtual screen (rows × cols of styled cells) — exactly what a real terminal would do, but as data.
  • osc133::Detector watches the same byte stream for prompt-marker escape sequences and emits PromptStart / CommandStart / CommandEnd events.
  • ratatui + crossterm own the real terminal: read keys, paint frames.
  • AppState is the single source of truth that ties them.

04Hosting the shell in a PTY

A pseudo-terminal is a kernel-provided pair of file descriptors that look like a terminal to the process on one end, and look like a pipe to the process on the other end.

The crate portable-pty abstracts macOS, Linux, and Windows. The spawn looks like this:

let pty = native_pty_system();
let pair = pty.openpty(PtySize { rows, cols, .. })?;

let mut cmd = CommandBuilder::new("/bin/zsh");
cmd.env("CMDQ_ACTIVE", "1");                    // integration knows it's inside cmdq
cmd.env("ZDOTDIR", &injected_zdotdir);            // see "Shell integration" below

let child  = pair.slave.spawn_command(cmd)?;
let reader = pair.master.try_clone_reader()?;       // shell bytes ←
let writer = pair.master.take_writer()?;            // → shell stdin

From here on, "writing a queued command" means: write its bytes into the master, exactly as if the user had typed them.

The PTY is the interface. Once you wrap something in a PTY, you don't care which shell it is. zsh, bash, fish, sh — they all just read keystrokes and write bytes. cmdq's queue, dispatch, and rendering are 100% shell-agnostic.

05OSC 133: knowing what the shell is doing

cmdq needs to answer a simple question on every keystroke: is the shell at a prompt, or is it running a command? That decides whether your keystroke goes through to the shell ("type ls and hit enter") or into the queue panel ("queue this for after the current job finishes").

Things that don't work

IdeaWhy not
Parse the prompt regex (PS1)Users have arbitrary prompts. Powerlevel10k, starship, custom — there's no universal pattern. False positives everywhere.
Watch the cursor positionHeuristic; breaks on every full-screen TUI, on multi-line prompts, on transient progress bars.
Poll ps for child processes of the shellRace-prone; expensive; misses shell builtins (the running "thing" isn't a separate process).
Trust a timeout"If no output for 200ms, assume prompt." Wrong on slow commands, wrong on tail-style watchers.

The right answer: the shell tells you

OSC 133 is a small de-facto-standard set of escape sequences that modern terminals (iTerm2, WezTerm, Kitty, Ghostty, VS Code) ask shells to emit, marking exactly the events we need. cmdq just listens for them.

The raw bytes look like this:

// printed by the shell, never displayed (terminals strip OSC sequences)
\x1b]133;A\x07          // before prompt
$ echo hi
\x1b]133;B\x07          // after prompt, before user types
\x1b]133;C\x07          // command starts running
hi
\x1b]133;D;0\x07        // command ended, exit 0

Streaming detector

The PTY hands cmdq bytes in arbitrary chunks — a marker can split across two reads. So the detector is a small state machine, not a regex:

enum State { Normal, AfterEsc, InOsc, InOscAfterEsc }

pub fn feed(&mut self, bytes: &[u8]) -> Vec<Event> {
    for &b in bytes {
        match self.state {
            State::Normal if b == 0x1b => self.state = State::AfterEsc,
            State::AfterEsc if b == b']' => { self.body.clear(); self.state = State::InOsc; }
            State::InOsc if b == 0x07 => { self.emit(out); self.state = State::Normal; }
            State::InOsc if b == 0x1b => self.state = State::InOscAfterEsc,
            State::InOscAfterEsc if b == b'\\' => { self.emit(out); self.state = State::Normal; }
            State::InOsc if self.body.len() < 256 => self.body.push(b),
            // ... reset on overflow / unrelated bytes
        }
    }
}

Two terminators are accepted: BEL (0x07) and ESC \ (the proper "string terminator"). The 256-byte body cap keeps a runaway emitter from growing the buffer unbounded. Twelve unit tests pin all the edge cases (split bytes, malformed bodies, mixed terminators, nested-looking sequences).

Lesson. When you need a fact that the controlled side already knows (here: "I'm at a prompt now"), don't infer it — get the controlled side to tell you. Inference is heuristics, telling is correctness.

06Rendering: vt100 inside ratatui

cmdq has to draw two things in one screen:

  1. The inner shell — colors, cursor, alt-screen TUIs, the works. This must look exactly like a real terminal.
  2. The queue panel below it — its own UI, with input line, queue list, hint line.

Doing the first one from scratch is a multi-year project (terminal compatibility is a swamp). Instead: let a terminal emulator crate keep an in-memory screen, then paint that screen into ratatui's frame.

The widget is short:

impl Widget for VtScreen<'_> {
    fn render(self, area: Rect, buf: &mut Buffer) {
        let screen = self.parser.screen();
        for row in 0..screen.size().0.min(area.height) {
            for col in 0..screen.size().1.min(area.width) {
                let cell = screen.cell(row, col).unwrap();
                let mut style = Style::default();
                if let Some(c) = vt_color_to_ratatui(cell.fgcolor()) { style = style.fg(c); }
                if cell.bold()      { style = style.add_modifier(Modifier::BOLD); }
                if cell.underline() { style = style.add_modifier(Modifier::UNDERLINED); }
                // ... italic, inverse, bg ...
                if let Some(c) = buf.cell_mut((area.x + col, area.y + row)) {
                    c.set_symbol(if cell.contents().is_empty() { " " } else { cell.contents() });
                    c.set_style(style);
                }
            }
        }
    }
}

07Queue model & dispatch

Just a Vec

pub struct QueueItem { pub id: u64, pub command: String, pub conditional: bool }
pub struct Queue     { items: Vec<QueueItem>, next_id: u64, pub paused: bool }

No fancy data structure — the queue is short by definition (it's a UI affordance, not a job system). Stable monotonic IDs make logging readable and editing safe across reorders.

Dispatch is "type the bytes for the user"

When a CommandEnd marker arrives, cmdq writes the next eligible item's bytes back into the PTY master:

if let Some(item) = state.queue.pop_next_eligible(exit_code) {
    writer.write_all(item.command.as_bytes())?;
    writer.write_all(b"\n")?;        // trigger the Enter
}

The shell can't tell the difference between "user typed it" and "cmdq wrote it" — that's the whole point of the PTY abstraction.

Conditional chaining

An item flagged conditional only runs if the previous command's exit code was 0. The exit code arrives inside the OSC 133 D marker (\x1b]133;D;1\x07). So conditional skipping isn't a separate machine — it's one filter on dispatch:

pub fn pop_next_eligible(&mut self, prev_exit: Option<i32>) -> Option<QueueItem> {
    while let Some(front) = self.items.first() {
        if front.conditional && prev_exit != Some(0) {
            self.items.remove(0);  // drop the skipped item, try the next
            continue;
        }
        return Some(self.items.remove(0));
    }
    None
}

Persistence

The queue is JSON-serialized to ~/.local/share/cmdq/queue.json so a crash mid-session doesn't lose pending work. Originally this saved on every event-loop tick (~60 Hz). After a code review pass it got a queue_dirty flag and now saves only on real mutation.

The bug that almost shipped. "Save the queue every iteration" feels safe — until you realize the loop runs at 60 Hz and you're rewriting the same JSON file 60 times per second of idle. The fix is a one-line dirty flag. The lesson: any unconditional work inside an event loop is suspect.

08Shell integration without surgery

For OSC 133 markers to exist at all, the shell has to emit them — so cmdq ships three tiny snippets (one per supported shell) that hook the right precmd/preexec points.

zsh

_cmdq_precmd()  { printf '\e]133;A\a'; }
_cmdq_preexec() { printf '\e]133;C\a'; }
precmd_functions+=(_cmdq_precmd)
preexec_functions+=(_cmdq_preexec)

bash

PROMPT_COMMAND for prompt, DEBUG trap for command-start, trap … EXIT for end. Bash makes you work for it.

fish

fish_preexec, fish_postexec, fish_prompt events. Cleanest of the three.

Each snippet is gated by $CMDQ_ACTIVE so it only runs inside cmdq, never in normal shell sessions.

The ZDOTDIR shim — auto-integration without editing user files

The first time you run cmdq, you don't want to be told "now go edit your .zshrc". You also don't want to silently edit it. So for zsh, cmdq builds an ephemeral ZDOTDIR:

09UX polish: alt-screen, SIGINT, discoverability

The basic mechanism (queue while running, dispatch on end) was working before any of this. These three problems only showed up in real use — and each one has a small, testable fix.

Problem 1: cmdq fights with TUI programs

You run vim. cmdq's queue panel is still showing at the bottom. Your arrow keys go into cmdq's editor instead of vim. Misery.

Solution: detect alt-screen. Full-screen TUIs (vim, htop, less, fzf, btop, nvim) all emit DECSET 1049 to switch into the alternate screen buffer. vt100::Parser exposes this as a single bool:

let on_alt = parser.screen().alternate_screen();
if on_alt && !state.auto_passthrough {
    state.auto_passthrough = true;       // hide panel, forward keys verbatim
} else if !on_alt && state.auto_passthrough {
    state.auto_passthrough = false;      // vim quit; back to queue mode
}

No user toggle. No config. No special list of programs. The signal is already in the byte stream — cmdq just listens.

Problem 2: Ctrl-C auto-fires the next command

You queue three things, change your mind on the running command, hit Ctrl-C — and cmdq dutifully fires the first queued command at you. That's exactly the wrong inference.

Solution: SIGINT auto-pause.

const SIGINT_AUTO_PAUSE_WINDOW: Duration = Duration::from_secs(3);

fn handle_command_end(state: &mut AppState, exit_code: Option<i32>, ..) {
    let recent_sigint = state.last_sigint_at
        .map(|t| t.elapsed() <= SIGINT_AUTO_PAUSE_WINDOW).unwrap_or(false);
    let exit_was_sigint = exit_code == Some(130);   // 128 + SIGINT

    if (recent_sigint || exit_was_sigint) && !state.queue.is_empty() {
        state.queue.paused = true;
        state.status = "Ctrl-C — queue paused. Ctrl-X to resume, Ctrl-K to clear.".into();
        return;
    }
    // ... else dispatch next eligible
}

Two signals are checked together: did we forward 0x03 to the PTY in the last 3 seconds, or did the marker report exit 130. Either is enough. Both are required because a misbehaved program might trap SIGINT and exit normally, or exit 130 from internal error.

Problem 3: the panel flashes for fast commands

Short commands like ls or cd finish in milliseconds. Surfacing the queue panel for them just makes the screen flicker every time you hit Enter at a prompt.

Solution: a 1.5-second grace period. The panel only appears once the current command has actually been running long enough to be worth queueing against. command_started_at is set on CommandStart, cleared on CommandEnd; in_queue_mode() only returns true once elapsed() >= QUEUE_PANEL_DELAY.

const QUEUE_PANEL_DELAY: Duration = Duration::from_millis(1500);

fn command_long_running(&self) -> bool {
    self.command_started_at
        .map(|t| t.elapsed() >= QUEUE_PANEL_DELAY)
        .unwrap_or(false)
}

fn in_queue_mode(&self) -> bool {
    !self.effective_passthrough() && (self.command_long_running() || self.force_queue)
}

Force-queue (Ctrl-Q) bypasses the delay: if you know you're about to need it, you can summon the panel immediately.

Problem 4: nobody knows the keybindings

A flashy TUI with twelve hidden shortcuts is hostile. Three-tier disclosure:

Tier 1 — always visible

A dim hint line at the bottom of the queue panel: ? help · Enter queue · ↑ edit · Tab cond · ⌃X pause · ⌃K clear · ⌃\ passthrough

Tier 2 — one keystroke

Press ? on an empty buffer or F1 anywhere — full overlay opens. Sections: queueing, queue control, modes, misc.

Tier 3 — README

Full table on disk for users skimming the project before they install.

10The event loop, end to end

One thread reads the PTY into an mpsc::channel; the main thread does everything else. No async runtime, no executor — the loop is short enough that cooperative polling is simpler than futures.

11Test strategy

51 tests across three layers. The point isn't the count — it's that each layer has tests written in its own vocabulary.

LayerWhat's testedHow
Unit (45)
fast		 OSC 133 detector edge cases · queue ops · line-editor key handling · conditional skipping · AppState toggles · SIGINT auto-pause logic Pure Rust, no I/O. cargo test --lib is < 50 ms.
Integration (4)
real shell		 Real bash spawned in a real PTY emits real OSC 133. cmdq's detector picks them up. Conditional-skip works end-to-end. portable-pty spawns /bin/bash --norc --rcfile <our integration> -i; reads bytes; matches against \x1b]133;D markers.
Binary smoke (2)
end-to-end		 The actual cmdq binary runs inside a PTY, force-queues a command, types a trigger command, observes both run. The crate exposes both a lib and a bin target so tests can drive the same binary users get.
Why bother with the binary smoke test? Because unit tests can't catch "I forgot to wire the OSC detector into the loop" or "ratatui crashes when alt-screen toggles too fast". The smoke test reproduces the whole runtime — terminal modes, signal handlers, panic hooks — and fails loudly if any of them are misconfigured.

12Rust-specific lessons

Crate choices that mattered

ConcernCrateWhy
Cross-platform PTYportable-ptyWezTerm's PTY layer; macOS/Linux/Windows in one API. The alternative (nix::pty + Windows ConPTY directly) is two implementations of the same thing.
Terminal raw mode + key parsingcrosstermStable, ergonomic, works on all three OSes. termion is Linux-only.
TUI renderingratatuiDiffing buffer-based renderer means you describe what to draw, it computes the minimal byte sequence to update the screen.
Terminal emulationvt100Maintains a virtual screen given byte input. Saves ~years of CSI/SGR/DECSET implementation work. Exposes alternate_screen(), which is how auto-passthrough is free.
Serializationserde + serde_jsonBoring on purpose. Queue persistence is a few hundred bytes.
CLI parsingclapDerive macros; three flags total.

Patterns worth keeping

AppState over function-arg sprawl

Original handle_key took 10 separate &mut parameters. After consolidation it takes &mut AppState. Adding a new piece of state is now a one-line struct field, not a refactor of every call site.

&'static str over String for static literals

input_prompt_prefix originally returned String with .to_string() in every match arm. Switching to &'static str dropped five allocations per render frame. Free.

Borrow in render hot paths

Span::raw(input_buffer) instead of Span::raw(input_buffer.to_string()). ratatui's Span<'a> is built for borrows; cloning was muscle-memory, not necessity.

mpsc::channel for the reader thread

One thread does read() on the PTY (a blocking call); it sends Vec<u8> chunks to the main thread via channel. Main thread try_recvs in a non-blocking loop. No async runtime needed for one producer + one consumer.

Streaming state machines beat regex

OSC 133 detection is a 4-state machine that handles arbitrary byte splits. A regex would buffer everything; the machine consumes one byte at a time and emits events as they complete.

Test the binary you ship

option_env!("CARGO_BIN_EXE_cmdq") gives integration tests the path to the real built binary. Running it inside portable-pty from a test catches startup/shutdown bugs that no unit test can.

Lib + bin in one crate

Adding [lib] alongside [[bin]] in Cargo.toml lets integration tests import internal modules and drive the compiled binary, with one toolchain run.

Dirty flags for persistence

"Save every loop iteration" → 60 disk writes per idle second. queue_dirty: bool, set on every mutation, cleared after save. Same correctness, ~zero idle I/O.

Panic hooks restore the terminal

If anything panics while raw mode is on, the user's terminal is wedged. std::panic::set_hook wrapping the original ensures alt-screen and raw mode are dropped before the panic surfaces.

Clippy as a forcing function

cargo clippy --all-targets -- -D warnings in CI catches collapsible_if, manual_repeat_n, needless_borrow, too_many_arguments — design smells masquerading as lints.

Things that surprised me

vt100 doesn't emit dirty regions. You walk every cell every frame. ratatui's own buffer diff (vs. previous frame) is what saves you — so you can be sloppy in the widget and still ship efficient terminal output. The two layers cover for each other.
Terminal cursor placement is separate from widget rendering. ratatui draws the frame; the OS cursor is a separate set_cursor_position call. cmdq has to decide every frame whether the cursor belongs in the queue input line or wherever vt100 says the inner shell put it.
--norc overrides --rcfile. Bash arg order matters. Spent an embarrassing amount of time on a test that "couldn't see" the integration script before discovering this.

13General software lessons

Move up a layer

If the layer you're trying to extend is fundamentally blocked from doing the thing you want (like a shell plugin during a foreground job), wrap it instead. tmux, debuggers, IDE terminals, containers — this is a recurring shape.

Don't infer what the source can tell you

Parsing PS1 regex is heuristic. Asking the shell to emit OSC 133 is correct. Whenever you find yourself writing detection logic, ask whether the system you're detecting could just notify instead.

Auto-detect > manual toggle

Alt-screen passthrough used to be Ctrl-\ only. Watching alternate_screen() made the right behavior the default; the manual toggle is now an escape hatch, not the primary UX.

Don't auto-fire after the user changed their mind

Ctrl-C is a strong "abort" signal. Treating it as "and then do the next thing in the queue" is technically correct and emotionally wrong. Detect and pause.

Three-tier disclosure for keybindings

Hint line (always visible) → overlay (one keypress away) → README (offline). Each tier has different verbosity. Beginners read tier 1; power users glance at tier 2; tire-kickers find tier 3.

Persist on mutation, not on tick

Anything you do unconditionally inside a 60 Hz loop is suspect. A boolean dirty flag is almost always right.

Separate poll cadence from render cadence

Input wants low latency (poll fast); rendering wants steady frame rate (gate). Conflating them — "redraw whenever anything changed" — produces both jitter and waste.

Idempotent install hooks

# >>> cmdq integration >>> / # <<< markers in user rc files mean reinstall is a no-op and uninstall is a delete-between-markers. Never assume your install will only run once.

Don't edit user dotfiles silently

Synthesize a temporary one (ZDOTDIR / BASH_ENV) for the in-process case; offer an explicit --install-integration for the durable case. Making the destructive path opt-in builds trust.

Test what you ship

If your test harness builds something different from your release artifact, your tests are testing a fiction. Use the cargo binary path for end-to-end tests.

Make state explicit

AppState with named fields beats ten &mut arguments. The struct documents the system; the arg list documents one function.

Two flags can mean two things

manual_passthrough + auto_passthrough looked redundant — but they encode different intent: "user asked for it" vs "program forced it". Combining them would make alt-screen exit clobber the user's deliberate choice.

Code review is a cheap pass

Three review agents (reuse, quality, efficiency) run in parallel found the per-tick disk write, dead code, and a handful of needless allocations. None were caught by tests; all were trivial to fix.

Keep escape hatches

Auto-passthrough is great — until it's wrong. Ctrl-Q for force-queue, Ctrl-\ for manual passthrough. The smart default doesn't have to be the only option.

UI that doesn't flicker is UI that disappears

The 1.5 s grace period before the queue panel appears removes a class of "wait, what just flashed?" interruptions for the 99% of commands that finish quickly. The right amount of UI for an instant command is none.
